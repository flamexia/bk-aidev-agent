# 标准作业流程 (SOP): 功能开发后的文档与日志更新

#### **目标**

确保每一次功能开发或代码变更后，项目文档和更新日志都能得到及时、准确、高质量的更新，保持代码与文档的同步，并为用户提供清晰的迭代记录。

#### **适用范围**

适用于任何新功能开发、功能增强、API 变更或重要修复。

#### **核心原则**

1.  **文档先行**: 在设计 API 时就考虑其文档友好性。
2.  **用户中心**: 从使用者的角度撰写文档和日志，解释"什么"和"为什么"，而不仅仅是"如何"。
3.  **渐进复杂性**: 文档结构应引导用户从基础入门到高级应用，平滑学习曲线。
4.  **一致性**: 确保所有相关文档（API、指南、FAQ、示例）的信息保持一致。
5.  **原子化**: 每个文档页面应聚焦于一个独立、清晰的主题。

---

### **第一阶段：Commit分析与变更识别**

在执行文档更新时，首先分析提供的变更信息并评估其对现有文档体系的影响。

1.  **获取变更信息**:
    - 如果提供commit范围：使用 `git diff` 或 `git show` 提取变更详情
    - 如果提供分支：使用 `git diff <branch>..<base-branch>` 获取差异
    - 如果提供功能描述：基于描述进行文档影响评估

2.  **变更分类**:
    - **API变更**: props、methods、events、types的增删改
    - **新功能**: 引入了全新的概念或能力 (例如: 编程式会话管理)。
    - **功能增强**: 在现有功能上进行扩展 (例如: 为已有 API 添加新参数)。
    - **API 破坏性变更**: 修改了现有 API 的签名或行为。
    - **修复**: 修正了 Bug。
    - **文档变更**: 现有文档的修改或更新

3.  **影响范围评估**:
    - 分析代码变更对API文档的影响
    - 识别需要更新的文档文件
    - 确定是否涉及破坏性变更
    - 评估变更的复杂性和关联性

4.  **文档策略决策**:
    - **简单增强/修复**: **更新**现有相关文档即可。
    - **独立新功能**: **创建**一个新的、独立的文档页面。
    - **复杂或关联性强的新功能**: **重构**现有文档结构。问自己："这个新功能是否让现有的文档结构变得混乱？" 如果是，就应该重新设计信息架构，采用更合理的组织方式（如"基础-进阶-高级"分层）。

---

### **第二阶段：文档内容实现**

根据第一阶段的策略，开始执行文档内容的编写和修改。

1.  **API 参考文档更新**:
    - `docs/api/props.md`: 添加/修改相关属性 (Props)。
    - `docs/api/methods.md`: 添加/修改相关方法 (Methods)。
    - `docs/api/events.md`: 添加/修改相关事件 (Events)。
    - `docs/api/types.md`: 添加/修改相关的 TypeScript 类型定义。
    - **要点**: 确保每个条目都有清晰的描述、类型、默认值和版本号标记（如 `<Badge type="tip" text="v1.2.2" />`）。

2.  **类型定义文件检查**:
    - 检查 `src/types/index.ts` 或类似类型定义文件，确认新功能的 TypeScript 类型定义已正确添加。
    - 确保类型定义与 API 文档保持一致。

3.  **指南文档更新/创建**:
    - **创建新页面**: 如果是新功能，在 `docs/guide/` 下的合适目录（如 `advanced-usage`）创建新的 `.md` 文件。
    - **提供真实示例**: 示例代码必须是可工作的、端到端的。**避免使用模拟数据或伪代码** (例如，将 `fetchSessionList` 从模拟实现更新为真实 API 调用)。
    - **解释核心概念**: 清晰地解释新功能的使用场景、核心概念和注意事项。

4.  **导航与侧边栏更新**:
    - 修改 `.vitepress/config.js` (或类似配置文件)。
    - 将新页面添加到侧边栏，并确保其在导航结构中的位置符合逻辑（例如，将新文档放入"编程交互"的层级菜单下）。

---

### **第三阶段：验证与链接修复**

这是最容易被忽略但至关重要的一步。

1.  **自动化测试**:
    - 运行文档构建命令验证无错误 (`pnpm run docs:build`)
    - 检查所有内部链接的有效性
    - 验证代码示例的语法正确性

2.  **一致性检查**:
    - 验证API文档与源码注释的一致性
    - 检查类型定义与文档描述的匹配性
    - 确认版本标记的准确性

3.  **全局搜索**: 在整个文档目录中，搜索被修改或删除的旧文件名（如 `programmatic-control.md`），找到所有引用它的地方。
4.  **修复死链**: 逐一修复所有找到的死链，将其指向新的、正确的文档路径。
    - 重点检查区域：
      - `api/` 目录下的交叉引用。
      - `demos/` 目录下的示例代码链接。
      - `faq.md` 中的解答链接。
      - `guide/` 目录下的快速上手和相关指南。
5.  **本地构建验证**: 运行文档站的本地构建命令 (`pnpm run dev` 或 `npm run docs:dev`)。VitePress 或其他现代文档工具会自动检测并报告所有死链。根据报错信息进行修复，直到构建成功且无警告。

---

### **第四阶段：更新日志 (Changelog) 管理**

1.  **智能版本推断**: 基于变更类型自动推断版本号（MAJOR.MINOR.PATCH）
    - 破坏性变更 → MAJOR版本
    - 新功能 → MINOR版本
    - 修复 → PATCH版本
2.  **结构化日志生成**:
    - 自动生成符合Conventional Commits格式的日志
    - 按类型分组（feat, fix, docs, perf等）
    - 包含相关的commit hash引用
3.  **起草更新内容**:
    - **特性 (Features)**: 清晰列出所有新增功能。
    - **改进 (Improvements/Enhancements)**: 描述所有优化和改进，如文档重构、性能提升等。
    - **修复 (Fixes)**: 列出所有被修复的 Bug。
    - **破坏性变更 (Breaking Changes)**: **【极其重要】** 单独、明确地列出所有可能影响用户现有代码的变更。
4.  **同步更新**:
    - 将起草好的日志条目，添加到项目根目录的 `CHANGELOG.md` 中。
    - 同时，将其添加到文档站点的 `docs/changelog.md` 文件中，确保两边内容同步。
    - 检查 `src/types/index.ts` 中的类型定义注释，确保版本标记（如 `@since v1.2.9`）与 changelog 中的版本一致。

### **使用说明**

要使用此文档更新流程，可以在 Claude Code 中执行以下命令：

支持以下使用方式：

1. 基于指定commit范围更新：
   ```
   /doc-update --from <commit-hash> --to <commit-hash>
   ```

2. 基于最近的commit更新：
   ```
   /doc-update --last <number-of-commits>
   ```

3. 基于分支差异更新：
   ```
   /doc-update --branch <branch-name>
   ```

4. 基于自定义描述更新：
   ```
   /doc-update [功能名称或简要描述]
   ```

例如：

```
/doc-update --last 3
/doc-update --from abc1234 --to def5678
/doc-update --branch feature/new-session-api
/doc-update 编程式会话管理
```

执行该命令后，Claude Code 将自动调用此子代理来处理完整的文档更新流程。

**重要提示**: 每次文档更新都应参考 `.claude/standard-doc-update-checklist.md` 中的标准检查清单，确保覆盖所有必要的文档文件和验证步骤，包括 API 文档、类型定义、指南文档和版本日志等。
